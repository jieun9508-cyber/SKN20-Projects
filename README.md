## 📌 제공하는 서비스

### Pseudo Practice
- **4단계 Progressive 학습 구조**
- Stage 1: 오염되거나 정제되지 않은 Raw Data 유입
- Stage 2: 4단계 사고 프로토콜(분석-설계-구현-승인) 적용
- Stage 3: re.sub(정규식), MSE(손실 측정), threshold(임계값) 등을 활용한 무결성 검증
- Stage 4: 정제된 데이터셋 또는 의사결정 신호 생성
- **AI 평가**: GPT-4o-mini 기반 5차원 역량 분석 (정합성, 추상화, 예외처리, 구현력, 설계력)

### Bug Hunt
- **실무 디버깅 시나리오 12 Cases**
- 데이터 관련 (Data Leakage, Label Imbalance, Overfitting)
- 코드 로직 (Off-by-one Error, Null Pointer, Type Mismatch)
- ML 특화 (Metric 선택 오류, Feature 누락, 하이퍼파라미터)
- **3단계 평가**:
  - Model A: 사고 방향 평가 (버그 원인 정확히 언급?)
  - Model B: 코드 위험 평가 (변경 라인 수, 조건문/예외 처리 변화)
  - Model C: 사고 연속 점수 (논리적 흐름, 근거 제시 품질)

### System Architecture
- **실제 시스템 설계 사례 기반 3가지 시나리오**
  - Image Feed (반려 식물 성장 일기): Low Latency, 확장성, 모니터링
  - Real-time Chat (1:1 멘토링 채팅): 대용량 처리, 확장성, 개인화
  - Newsfeed (동아리 통합 공지): 대용량 처리, 확장성, A/B Testing
- **2-Phase 평가**:
  - Phase 1: 아키텍처 설계 평가 (필수 컴포넌트, 연결 관계, Mermaid 다이어그램)
  - Phase 2: Deep Dive 면접 (설계 의도, 확장성/성능, 장애 대응)

## 📊 평가 시스템

### Pseudo Practice - AI 기반 5차원 메트릭 평가

**평가 철학**: 정답 채점 ❌ → 사고력 평가 ✅

| 평가 항목 | 설명 | 배점 |
| --- | --- | --- |
| **정합성** | quest_title과 사용자 로직의 일치 여부 확인 | 0-100점 |
| **추상화** | 의사코드의 간결성 및 핵심 로직 표현력 평가 | 0-100점 |
| **예외처리** | 예외 상황 대응 키워드 및 로직 포함 여부 분석 | 0-100점 |
| **구현력** | 의사코드의 구체성과 논리적 흐름의 명확성 평가 | 0-100점 |
| **설계력** | 단계별 연결성 및 아키텍처적 완성도 검증 | 0-100점 |

**특징**:
- 논리적 연결성 검증 (단순 키워드 나열 → 40점 이하)
- Tail Question: 논리적 허점 질문 자동 생성
- 교육적 피드백: 시니어 엔지니어 관점 조언

### Bug Hunt - 3단계 디버깅 사고력 평가

**평가 대상**: 정오답 ❌ → 디버깅 사고의 질 ✅

| Model | 평가 내용 | 출력 |
| --- | --- | --- |
| **Model A: 사고 방향 평가** | 버그 원인 정확히 언급?<br>원인-수정 일치? 부작용 고려?<br>설명-코드 일관성? | 통과/실패 |
| **Model B: 코드 위험 평가** | 변경 라인 수 분석<br>조건문/예외 처리 변화<br>기존 로직 훼손 여부 | 위험도 (0~100) |
| **Model C: 사고 연속 점수** | 논리적 흐름 분석<br>근거 제시 품질<br>기술적 정확성 | 사고력 (0~100) |

**Step별 개별 피드백**:
- Step 1, 2, 3 각각에 대한 설명 품질 점수
- 잘한 점 / 부족한 점 / 개선 방향 구체적 제시
- 총평: 디버깅 방향성 평가 + 전반적 조언

### System Architecture - 2-Phase 평가

| Phase | 평가 내용 |
| --- | --- |
| **Phase 1: 아키텍처 설계 평가** | • 필수 컴포넌트 포함 여부 (Frontend, Backend, DB, ML Model...)<br>• 컴포넌트 간 연결 관계 적절성<br>• Mermaid 다이어그램 |
| **Phase 2: Deep Dive 면접** | AI가 3개 카테고리 질문 자동 생성:<br>• 설계 의도: 왜 이런 구조를 선택?<br>• 확장성/성능: 트래픽 10배 증가 시?<br>• 장애 대응: DB 장애 시 동작 방식? |

**평가 결과**: 설계 점수 + 면접 점수 = 총점 | 설계 피드백 + 면접 피드백 + 개선 제안

## 🔄 PoC 주요 개선 과제

### 1. DB 구조
- **현재**: Practice/User/UserDetail 중심의 단순 구조, 일부 로컬 JSON과 병행 사용
- **문제점**: 데이터 소스 이원화로 확장/버전관리 어려움
- **해결 방안**: 단일 User 모델로 통합, DB 통합 스키마로 이동

### 2. 기능 확장
- **현재**: 3개의 Practice 완료, Ops/Agent Practice 미완료
- **문제점**: 신규 Practice 추가 시 UI/데이터 모두 수정 필요
- **해결 방안**: Ops, Agent Practice 구현, 백엔드에 진도 저장

### 3. AI 코치, 평가 성능
- **현재**: LLM 프롬프트 기반 단순 평가 방식
- **문제점**: 평가 일관성 부족(±5점 편차), 환각 발생, 응답 속도 지연
- **해결 방안**:
  - Fine-tuning 적용 (평가 일관성 확보 ±5점 → ±2점)
  - 체계적 루브릭 설계
  - 정량/정성 평가 분리
  - 실시간 피드백

### 4. 보안
- **현재**: AI API 노출 위험, 본인 체크 없음
- **문제점**: 키 유출/오용 가능성, 프롬프트 인젝션/응답 위변조 가능
- **해결 방안**:
  - 사용자 인증 필수화
  - Rate Limiting 적용
  - API 키 백엔드로 이동
