## 슬라이드 1 | 전체 시스템 흐름

### 시스템 아키텍처 Practice 전체 흐름

```
[문제 선택]  DB API → Junior 10문제 / Advanced 31문제
      ↓
[캔버스 설계]  드래그&드롭 컴포넌트 배치 + 연결
      ↓
[클라이언트 사전 검증]  ← LLM 호출 전 필터링
    1단계: 기본 구조 (컴포넌트 3개↑, 연결 존재, 고립 없음)
    2단계: 필수 요구사항 (required_components 70%↑, required_flows)
    3단계: 설계 품질 (타입 다양성 — 경고만)
      ↓ 통과
[설명 입력]  최소 30자 + 무의미 답변 필터링
      ↓
[심화질문 생성]  안티패턴 기반, 문제별 Pillar 동적 선별 → 3개 질문 (gpt-5-mini)
      ↓
[심화질문 답변]  최소 10자 + 단순 반응어 차단
      ↓
[LLM 평가]  GCP 6필라 × 문제별 가중치 (gpt-4o-mini)
      ↓
[결과 저장]  DB 영속 (UserSolvedProblem)
```

## 슬라이드 2 | 개선 ① — 문제 리스트

### 문제 데이터의 출처 신뢰도 확보

| | 중간 발표 | 최종 발표 |
|---|---|---|
| 출처 | 수동 작성, 출처 불명확 | 실전 인터뷰 사이트 크롤링 기반 |
| 문제 수 | 10문제 | **31문제 (+210%)** |

**접근 방식**
- `interviewing.io/system-design-interview-questions` 크롤링
- 실제 빅테크 시스템 디자인 인터뷰 문항 데이터 기반으로 LLM 가공
- Amazon 판매 랭킹, 은행 원장, 실시간 댓글, 온라인 저지, Robinhood 등

> 문제 자체가 실전 인터뷰에서 출제되는 유형인지 검증되지 않으면 평가 신뢰도도 의미가 없음 → 출처부터 확보

## 슬라이드 3 | 개선 ② — 평가 기준 고도화 (GCP 6필라)

### LLM 단독 판단 → 구조화된 루브릭 평가

**AS-IS**: LLM이 자체 판단으로 점수 부여 → 기준 불명확, 일관성 부족

**TO-BE**: Google Cloud Well-Architected Framework 6필라 기반 자체 평가 기준 설계

| 필라 | 설명 | 적용 예 |
|---|---|---|
| Operational Excellence | 운영 자동화, 모니터링 | 배치 파이프라인 운영 |
| Security | 접근 제어, 규제 준수 | MFA, 감사 로그 |
| Reliability | 일관성, 장애 복구 | Strong Consistency |
| Cost Optimization | 비용 효율 구조 | 핫/콜드 데이터 분리 |
| Performance Optimization | 처리량, 응답 지연 | 캐시 샤딩 전략 |
| Sustainability | 자원 효율, 에너지 | 배치 최적화 |

- 문제마다 가중치를 다르게 부여 (문제 특성 반영)
  - 예) 은행 원장: Security(30) + Reliability(30) 최우선
  - 예) Amazon 랭킹: Performance Optimization(30) 최우선
- `reason` 필드로 가중치 선정 근거 명시 → 평가 기준 합리성 확보
- ※ 수치의 절대값보다 상대적 우선순위가 핵심

## 슬라이드 4 | 개선 ② — 평가 신뢰성 검증 (7모델 벤치마크)

### 7개 모델 × 420회 평가로 LLM 신뢰성 수치화

| 순위 | 모델 | 통과 | 일관성 σ | 구분력 | 순위 τ | 타당도 r | 비용 |
|:---:|---|:---:|:---:|:---:|:---:|:---:|:---:|
| 1위 | gpt-5.2 | 4/4 | 4.06 ✅ | 73.8pt ✅ | 0.850 ✅ | 0.722 ✅ | $3.70 |
| 2위 | gemini-2.5-pro | 3/4 | 4.80 ✅ | 62.2pt ✅ | 0.900 ✅ | 0.562 ❌ | $0.30 |
| 3위 | gpt-4o | 3/4 | 1.96 ✅ | 37.7pt ✅ | 0.800 ✅ | 0.610 ❌ | $1.30 |
| 5위 | gpt-5-mini | 3/4 | 2.90 ✅ | 32.7pt ✅ | 0.950 ✅ | 0.637 ❌ | $0.65 |
| 현재 | gpt-4o-mini | 2/4 | 2.17 ✅ | 27.7pt ❌ | 0.850 ✅ | 0.627 ❌ | $0.08 |

**(1) gpt-4o-mini를 선택한 이유**

아키텍처 평가 특성상 일관성·순위 정확도가 최우선 — 이 두 지표에서 gpt-4o-mini는 7모델 중 최상위권
- 일관성 σ=2.17 (최상위), 순위 τ=0.850 (기준 통과), 신뢰도 100%
- 비용 $0.08 — gpt-5.2 대비 46배 저렴

**(2) 한계 + 보완 방향**

| 미달 지표 | 수치 | 보완 방향 |
|---|---|---|
| 구분력 | 27.7pt (목표 30pt) | 점수 스케일링 조절 / 룰 앵커 강화 |
| 수렴 타당도 | r=0.627 (목표 0.65) | 룰 기반 비중 확대 |

> 모두 모델 교체 없이 프롬프트·루브릭 조정만으로 적용 가능

## 슬라이드 5 | 개선 ③④ — 클라이언트 검증 + 심화질문

### ③ 제출 전 3단계 사전 검증 (LLM 비용 절감)

불완전 제출도 즉시 LLM 호출 → 클라이언트에서 먼저 필터링

```
1단계 BLOCK  : 기본 구조 — 컴포넌트 3개↑, 연결 존재, 고립 컴포넌트 없음
2단계 BLOCK  : 필수 요구사항 — required_components 70%↑, required_flows 충족
3단계 WARNING: 설계 품질 — 타입 다양성 부족 시 경고 (통과는 가능)
```

답변도 LLM 전 클라이언트 검증: 설명 최소 30자 / 심화답변 최소 10자, 무의미 패턴 차단

---

### ④ 안티패턴 기반 심화질문 생성 (모델 상향)

범용 템플릿 질문 → 설계에 맞춘 날카로운 질문으로

```
지원자 설계 분석
  → 안티패턴 체크 (SPOF / Auto Scaling 부재 / Public DB 노출 등)
  → 문제 시나리오 기반 Pillar 동적 선별
  → 장애 시나리오 형태 개방형 질문 3개 생성
     예) "○○ 서버가 다운되면 자동 장애조치가 작동하나요?"
     예) "동시 사용자가 50배 급증할 때 이 아키텍처가 자동 대응하나요?"
```

- 모델: `gpt-4o-mini` → `gpt-5-mini` 상향 → 질문 구체성·품질 향상
- 카테고리(신뢰성/성능/운영) 고정으로 일관성 확보

## 슬라이드 6 | 종합 — 개선 전후 비교

### 시스템 아키텍처 Practice 중간 → 최종 개선 요약

| 영역 | 중간 발표 | 최종 발표 |
|---|---|---|
| 문제 출처 | 수동 작성 10문제 | 실전 인터뷰 크롤링 + LLM 가공 31문제 |
| 평가 기준 | LLM 자체 판단 | GCP 6필라 × 문제별 가중치 + reason 명시 |
| 검증 체계 | 없음 (즉시 LLM 호출) | 클라이언트 3단계 사전 검증 후 LLM 호출 |
| 심화질문 | 범용 템플릿 | 안티패턴 기반 + 상위 모델(gpt-5-mini) |
| 신뢰성 검증 | 없음 | 7모델 420회 벤치마크 — 수치로 검증 |

**평가 신뢰성 현황**
- gpt-4o-mini 기준 4개 지표 중 2개 통과 (일관성·순위)
- 미달 2개(구분력·타당도)는 원인 분석 완료, 모델 교체 없이 보완 가능
- 교체 시 최적 모델: gpt-5.2 (유일한 4/4 완전 통과)
