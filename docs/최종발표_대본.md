## 01️⃣ 기술 스택 & 시스템 아키텍처 (약 2분)

### 백엔드 (Backend)

| 분류 | 기술 |
|------|------|
| **Framework** | Django + Django REST Framework |
| **Database** | PostgreSQL (Supabase) |
| **Real-time Communication** | Socket.IO (WebSocket) |
| **AI/LLM** | GPT-4o, GPT-4o-mini, OpenAI Whisper |
| **Code Execution** | Docker Sandbox (안전한 격리 실행) |
| **Agent Framework** | LangGraph, LangChain |

### 프론트엔드 (Frontend)

| 분류 | 기술 |
|------|------|
| **Framework** | Vue 3 (Composition API) + Vite |
| **State Management** | Pinia |
| **Styling** | Tailwind CSS |
| **Real-time** | Socket.IO-client |
| **Visualization** | Chart.js, Mermaid |
| **Vision** | Google MediaPipe Tasks Vision |

### 시스템 아키텍처

```
클라이언트 (Vue 3)
    ↓ REST API + WebSocket
서버 (Django + DRF)
    ↙ ↓ ↘
PostgreSQL  GPT APIs   Docker Sandbox
(Supabase) (LLM)      (Code Execution)
```

### 핵심 원칙: 6개 기능 × 6개의 다른 AI 설계

같은 OpenAI API를 쓰지만, **기능마다 완전히 다른 AI 아키텍처**:

| 기능 | AI 아키텍처 | 핵심 특징 |
|------|------------|----------|
| **Pseudocode** | 뉴로심볼릭 (LLM + Rule) | LLM 85점 + Rule 감점 15점 |
| **Bug Hunt** | 4단계 순차 파이프라인 | 코드검증 → 인터뷰 → 루브릭 → 종합 |
| **Architecture** | 6개 Pillar 평가 엔진 | AWS 표준으로 설계 평가 |
| **Wars** | LangGraph 4-Agent | Coach/Chaos/Eval Agent 오케스트레이션 |
| **AI Coach** | ReAct Agent | 6개 도구 자율 호출 |
| **Mock Interview** | 5-Layer 면접 엔진 | 3개 Layer는 Rule, 2개는 LLM |

### 구현 전략

**6개 기능 × 독립적 모듈화:**

각 Practice 타입을 독립적 모듈로 구현하되, 공통 기능은 계층화:

```
/backend/core/
├── models/              # DB 모델 정의
├── views/               # API 엔드포인트 (8개 Practice별)
├── services/            # 비즈니스 로직 (AI 평가, 코칭)
├── agents/              # LangGraph Agent 파이프라인
├── utils/               # 공통 유틸리티 (샌드박스, LLM)
└── middleware.py        # 인증, 로깅, CORS
```

### 🎓 배운 점 (Lessons Learned)

#### 1. AI 평가 엔진의 복잡성
- **LLM은 강력하지만 일관성 보증 어려움**

#### 2. 실시간 통신의 복잡성
- **Socket.IO 상태 관리 어려움**
- **해결책:** 룸 기반 격리, 명확한 이벤트 명세

#### 3. DB 설계의 중요성
- **초기 JSON 의존 → 성능 저하**
- **해결책:** 정규화된 스키마로 마이그레이션

#### 4. 개발 초반 요구사항 명확화
- **명확하지 않은 요구사항 → 리팩토링 비용 폭증**
- **해결책:** Agile 스프린트로 빠른 피드백

#### 5. 보안은 사후 처리 아님
- **초기부터 의존도 높음**
- **해결책:** 다음 프로젝트부터 OWASP Top 10 체크리스트 활용

### 5분 데모 시나리오 (발표 보조용)

1. **Pseudocode** (1분)
   - 의사코드 제출 → 5차원 뉴로심볼릭 평가 + 안티패턴 감지

2. **Bug Hunt** (1.5분)
   - 버그 수정 → 코드 검증 → 심층 인터뷰 (SSE 스트리밍)

3. **System Architecture** (1.5분)
   - DnD 설계 → Mermaid 자동 생성 → 6-Pillar 평가

4. **Mock Interview** (1분)
   - Job Planner 채용공고 파싱 → 맞춤형 면접 (음성)

---

**발표 전체 예상 시간:**
- 본 발표: 약 15분
- Q&A: 약 5분
- 데모: 약 5분 (선택사항)
- **총합: 약 25분**
>
> 단순 채점이 아닙니다. **5차원 AI 평가**입니다.
> 설계력 25점, 정합성 20점, 추상화 15점, 예외처리 15점, 구현력 10점.
> 이걸 LLM 85점과 Rule 기반 감점 15점을 결합한 **뉴로심볼릭 하이브리드**로 평가합니다.
>
> Rule 15점이 앵커 역할을 합니다.
> Hardcoding을 감지하면 -5점, Data Leakage를 감지하면 -5점.
> LLM이 흔들려도 Rule이 바닥을 잡아줍니다.
> 반대로, 복붙이나 무성의한 입력은 LowEffortDetector가 잡아서 0점 처리합니다.
>
> 그리고 점수대별로 다른 꼬리질문이 자동 생성됩니다.
> 고득점자에게는 실무 트레이드오프 질문, 저득점자에게는 힌트와 오해 교정 질문이 나옵니다.
>
> 검증 결과: 150회 반복 평가에서 **4개 지표 전부 통과**했습니다.
> 일관성 sigma 2.12, 구분력 61.1pt, 순위정확도 tau 0.867, 수렴타당도 r 0.751.

### 슬라이드 9-13: Bug Hunt (1.5분)

> 두 번째, Bug Hunt입니다. 정답을 맞히는 게 아니라 **디버깅 사고 과정을 평가**합니다.
>
> 4단계 순차 파이프라인입니다.
> Stage 1에서 코드를 검증하고, Stage 2에서 gpt-5.2가 심층 인터뷰를 합니다.
> "왜 고쳤는지" 물어봅니다. 여기서 진짜 실력이 드러납니다.
> Stage 3에서 3-Tier 루브릭으로 채점하고, Stage 4에서 gpt-5-mini가 종합합니다.
>
> 핵심은 이겁니다. Stage 1을 둘 다 Pass인데 최종 점수가 다릅니다.
> 코드를 고치는 것과 이해하는 것을 분리해서 평가하기 때문입니다.
>
> 검증 과정도 말씀드리겠습니다.
> 처음부터 완벽하지 않았습니다. 1단계에서 구분력은 통과했지만 일관성이 불안정했고,
> 2단계에서 프롬프트를 개선했더니 오히려 일관성이 악화됐습니다.
> 결국 **시스템 아키텍처 자체를 재설계**해서 3단계에서 완전 통과했습니다.
> 구분력 82.4pt, Kendall tau 0.983.
>
> 또한 7개 모델을 1,080회 벤치마크해서 **"비싼 모델이 좋은 게 아니다"**를 데이터로 확인했습니다.
> gpt-5는 5위, $5.10이었고, gpt-5.2가 1위, $2.50이었습니다.

### 슬라이드 14-18: System Architecture (1분)

> 세 번째, System Architecture Practice입니다.
> 드래그앤드롭으로 설계하면, **AWS Well-Architected 6-Pillar 표준으로 평가**합니다.
>
> 캔버스에서 컴포넌트를 배치하고 화살표로 연결하면,
> 이걸 JSON으로 직렬화하고 Mermaid 다이어그램으로 자동 변환한 다음,
> LLM이 6개 축 — 신뢰성, 성능, 운영우수성, 비용최적화, 보안, 지속가능성 — 으로 평가합니다.
>
> Anti-Pattern도 자동 감지합니다. SPOF, No Redundancy, Single AZ 등 8종.
> DB 1대만 배치하면 "SPOF detected" 경고가 뜨고 점수가 낮아집니다.
>
> 총 41종 시나리오 — 주니어 10종, 심화 31종 GCP 기반 — 를 준비했습니다.
> 검증은 100회 평가에서 4개 지표 중 3개 통과.
> 구분력만 28.9pt로 목표 30pt에 1.1pt 미달했는데,
> 이건 6개 축이 범용 기준이라 very_poor도 fair 점수를 받는 구조적 한계입니다.
> **문제를 숨기지 않고 분석한 것 자체가 전문성의 증거**라고 생각합니다.

### 슬라이드 19-23: Battle Game (1.5분)

> 네 번째, Battle Game입니다. **"상대 앞에서 실력이 드러난다"**.
> 1:1 실시간 대전에 AI 에이전트가 개입합니다.
>
> LangGraph 기반 4-Agent 오케스트레이션입니다.
> OrchestratorAgent가 게임 상태를 관찰하고, 조건에 따라 세 에이전트를 호출합니다.
>
> **CoachAgent**는 3단계로 에스컬레이션합니다.
> 첫 힌트는 Rule 기반으로 비용 0, 일관성 보장.
> 무시하면 LLM으로 소크라테스식 사고 유도.
> 그래도 안 되면 LLM으로 단계별 가이드. 쿨다운 40초로 무분별한 개입 방지.
>
> **ChaosAgent**는 배치된 노드를 분석해서 취약점을 노립니다.
> DB 1대만 배치하면 DB 장애 이벤트가 발생합니다.
> 무작위가 아닙니다. LLM이 생성한 이벤트를 Rule로 3가지 검증하고,
> 실패하면 사전 정의 이벤트로 폴백. 라운드당 1회만.
>
> **EvalAgent**는 양측 설계를 비교 평가한 뒤, self_critique로 편향을 자체 검증합니다.
> "REVISE" 판정 시 최대 2회 재심. 서버 권위 점수 우선으로 조작 불가.
>
> 미니게임은 3종입니다.
> 아키텍처 캐치마인드, 로직 런, 버그버블 몬스터.
> Socket.IO 기반 20개 이상 이벤트로 실시간 동기화됩니다.

### 슬라이드 24-26: AI Coach (1분)

> 다섯 번째, AI Coach입니다. 단순 챗봇이 아니라 **Agent**입니다.
>
> 3단계 파이프라인입니다.
> 먼저 Guardrail이 비학습 입력을 토큰 0으로 차단합니다. "오늘 점심 뭐 먹지?" 같은 질문이요.
> 다음 gpt-4o-mini가 Intent를 A부터 G까지 7가지로 분류합니다.
> 마지막으로 gpt-5-mini Agent가 6개 도구를 최대 5회 자율 호출합니다.
>
> 도구는 성적조회, 약점분석, 최근활동, 문제추천, 커리큘럼, 학습가이드.
> 중요한 건 Intent별로 사용 가능한 도구가 사전 제한되어 있어서 헛된 호출을 방지합니다.
> 도구 결과는 캐싱해서 같은 Agent Loop 안에서 중복 호출을 제거합니다.
